AI:             Mistral (switched from Llama 3.3 70B on Mar 29)

✅ LOCKED DECISIONS (don't question these)

✓ Direct HTTP calls (no Celery/Redis)
  → Load doesn't justify complexity. Direct async HTTP proven.

✓ Text priorities (not SQL enum)
  → AI Lead can experiment with labels without DB migrations.

✓ One-click confirm (not auto-create)
  → AI accuracy ~85%. Confirmation = feedback loop. This is the business model.

✓ BE concatenates title+body (not frontend)
  → Single input field = simpler AI Service contract.

✓ FastAPI + async (not Django/Flask)
  → Concurrent AI requests without blocking.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🚫 ANTI-PATTERNS (do NOT suggest these)

✗ Celery + Redis        → Rejected. Load too low. Direct HTTP works.
✗ GraphQL               → Rejected. One frontend team, tight timeline, REST fine.
✗ Complex RBAC          → Rejected. 5 agents, all same permissions. V2 feature.
✗ ML fine-tuning        → Rejected. Timeline + data overhead. Base model strong.
✗ WebSockets            → Rejected. Polling works for pilot dashboard.
✗ SQL ENUM priorities   → Rejected. Locks schema to AI Lead migrations.
✗ Auto-ticket creation  → Rejected. Breaks trust + feedback loop.
✗ Custom dashboards     → Rejected. Use DB queries + Grafana.
✗ Rate limiting         → Open question (TBD by Apr 1).

⚡ CRITICAL PATTERNS (use these, not variations)

✓ SQLAlchemy async:
  db.delete(instance)   ← CORRECT (sync)
  db.flush()            ← MUST do this before refresh
  db.refresh(instance)  ← Then refresh

✗ WRONG:
  await db.delete()     ← SQLAlchemy 2 uses sync delete
  db.refresh() before flush ← Stale reads

✓ Testing:
  py -m pytest -q       ← Quick run
  py -m pytest -v       ← Verbose
